# skillcheck v0.2 Build Spec

Principal-engineer synthesis of research + verification, 2026-07-06. Two research corrections are baked in (both verification REFUTED items): the name-charset error message ends with "**are** allowed." (not "allowed."), and the unknown-fields message has a **period** before "Only" (not a comma). Parity target is pinned to **skills-ref 0.1.0**.

## 1. RULE INVENTORY v0.2

Code scheme: `SC` prefix, numeric ranges by category (SC0xx frontmatter, SC1xx structure/references, SC2xx body, SC3xx extensions/portability). Every code has an interchangeable kebab-case alias. Codes are never renumbered or recycled. v0.1's 16 rules are retro-assigned codes here; verify against the shipped v0.1 rule table when coding, behavior corrections are flagged.

Severity tiers: **error** = skills-ref 0.1.0 strict parity, **warning** = beyond-parity checks + client-guide lenient downgrades, **info** = advisory heuristics that never gate CI. `--profile lenient` downgrades SC009 and SC014 to warning (per the client-guide contract: name-too-long and dir-mismatch are "warn, load anyway"; missing description and broken YAML stay fatal in both profiles).

### Frontmatter (SC0xx)

| Code | Alias | Sev | Origin | Check logic |
|---|---|---|---|---|
| SC001 | skill-md-missing | error | v0.1 | Path exists, is a directory, contains `SKILL.md` or `skill.md` (try uppercase first, matching `find_skill_md`). Messages: `Path does not exist: {dir}`, `Not a directory: {dir}`, `Missing required file: SKILL.md`. Accept a direct SKILL.md file path and resolve to parent dir. |
| SC002 | frontmatter-missing | error | v0.1 | Content must start with `---`. Message: `SKILL.md must start with YAML frontmatter (---)`. |
| SC003 | frontmatter-unclosed | error | v0.1 | Replicate `content.split("---", 2)`; fewer than 3 parts -> `SKILL.md frontmatter not properly closed with ---`. NOTE: split-based, so a literal `---` inside the YAML terminates it early. Replicate exactly for parity; do NOT switch to line-anchored `/^---$/`. |
| SC004 | frontmatter-invalid-yaml | error | v0.1, tightened | Parse YAML. **v0.2 change:** enforce strictyaml semantics on top of the JS parser -- reject flow-style collections (`{a: b}`, `[x, y]`), YAML tags, and duplicate keys, since skills-ref uses strictyaml and js-yaml/yaml is laxer. Message prefix: `Invalid YAML in frontmatter: {detail}`. |
| SC005 | frontmatter-not-mapping | error | v0.1 | Parsed result must be a mapping. Message: `SKILL.md frontmatter must be a YAML mapping`. |
| SC006 | name-missing | error | v0.1 | `name` key absent -> `Missing required field in frontmatter: name`. |
| SC007 | description-missing | error | v0.1 | `description` key absent -> `Missing required field in frontmatter: description`. |
| SC008 | name-empty | error | v0.1 | Not a string, empty, or whitespace-only -> `Field 'name' must be a non-empty string`. Early-return (skip SC009-SC014). |
| SC009 | name-too-long | error (lenient: warning) | v0.1 | After `name.trim()` + **NFKC normalize**, length > 64 -> `Skill name '{name}' exceeds 64 character limit ({len} chars)`. |
| SC010 | name-not-lowercase | error | v0.1, **corrected** | `name !== name.toLowerCase()` using Unicode-aware toLowerCase on the NFKC-normalized name -> `Skill name '{name}' must be lowercase`. Uppercase Cyrillic must fail; lowercase Cyrillic must pass. |
| SC011 | name-edge-hyphen | error | v0.1 | Starts or ends with `-` -> `Skill name cannot start or end with a hyphen`. |
| SC012 | name-consecutive-hyphens | error | v0.1 | Contains `--` -> `Skill name cannot contain consecutive hyphens`. |
| SC013 | name-invalid-chars | error | v0.1, **corrected** | Every char must be Unicode-alphanumeric or `-`. JS equivalent of Python `str.isalnum()`: `/^[\p{L}\p{N}\p{Nl}\p{No}-]*$/u` (add unit tests against skills-ref i18n fixtures: `技能`, `навык`, `мой-навык` pass; `skill_name` fails). **Do NOT use `/^[a-z0-9-]+$/` -- that breaks i18n parity.** Message (verified exact): `Skill name '{name}' contains invalid characters. Only letters, digits, and hyphens are allowed.` |
| SC014 | name-dir-mismatch | error (lenient: warning) | v0.1 | NFKC-normalize BOTH the name and the directory name, compare -> `Directory name '{dir}' must match skill name '{name}'`. Composed vs decomposed `café` must NOT error. Skip when linting a bare file with no meaningful parent. |
| SC015 | description-empty | error | v0.1 | Not a string, empty, or whitespace-only -> `Field 'description' must be a non-empty string`. |
| SC016 | description-too-long | error | v0.1 | Length > 1024 -> `Description exceeds 1024 character limit ({len} chars)`. |
| SC017 | compatibility-invalid | error | v0.1 | Not a string -> `Field 'compatibility' must be a string`; length > 500 -> `Compatibility exceeds 500 character limit ({len} chars)`. |
| SC018 | unknown-frontmatter-field | error | v0.1, **message corrected + rerouted** | Key outside spec set `{name, description, license, allowed-tools, metadata, compatibility}` AND outside KNOWN_EXTENSIONS -> error. Message (verified exact, period before Only): `Unexpected fields in frontmatter: {extras}. Only {allowed} are allowed.` (extras `', '`-joined sorted; allowed rendered as sorted list). Keys that ARE in KNOWN_EXTENSIONS route to SC301 instead (see below) -- otherwise every Claude Code skill errors. |
| SC019 | metadata-not-string-map | warning | **new** | `metadata` present but not a map, or values not scalar-coercible to strings. skills-ref coerces `{str(k): str(v)}`; spec says string-to-string map. |
| SC020 | skill-md-lowercase-filename | warning | **new** | `skill.md` found instead of `SKILL.md`. Valid per skills-ref, non-canonical. |
| SC021 | unquoted-colon-description | warning | **new** | Raw frontmatter line for `description` (or `name`) contains an unquoted value with `: ` (the client-guide "Use this skill when: ..." break case). Fix-it suggestion: wrap in quotes or block scalar. Fires from raw-text scan even when the parser recovered. |

| Code | Alias | Sev | Check logic |
|---|---|---|---|
| SC101 | broken-relative-reference | warning | Extract relative paths from the body: markdown link targets, plus inline-code tokens matching `(scripts\|references\|assets)/[^\s)`"']+`. Resolve against the skill dir (parent of SKILL.md). Missing file -> warning. Skip URLs, anchors, absolute paths (SC102's job), template placeholders (`${...}`, `{{...}}`). |

## 2. KNOWN_EXTENSIONS TABLE

Ship this as a typed registry (`src/extensions.ts`), driving SC301/SC302. Spec-official fields excluded (they are ALLOWED_FIELDS).

| Field | Type | Allowed values | Runtimes |
|---|---|---|---|
| `when_to_use` | string | free text | Claude Code only |
| `argument-hint` | string | free text | Claude Code, Copilot/VS Code |
| `arguments` | string \| string[] | arg names | Claude Code only |
| `disable-model-invocation` | boolean | true/false (default false) | Claude Code, Cursor, Copilot/VS Code |

## 3. EVALS.JSON FORMAT

**Alignment: strict superset of Anthropic skill-creator's `evals/evals.json`** -- the de-facto interchange format (nkootstra/skill-eval targets it; adewale/skill-eval-harness consumes its trigger rows). Anything skill-creator writes, skillcheck validates unchanged. skillcheck adds one thing skill-creator keeps in a separate file: a `triggers` array, fusing Skill-Lab's 4-type taxonomy with skill-creator's `should_trigger` polarity.

v0.2 ships two subcommands, **no runtime execution**:

- `skillcheck eval init <skill-dir>` -- scaffolds `evals/evals.json` (skill_name prefilled, one example eval, 13-slot trigger skeleton: 3 explicit / 3 implicit / 3 contextual / 4 negative, with >= 2 same-domain near-miss negatives called out in comments-via-placeholder-text).
- `skillcheck eval check <skill-dir>` -- validates the file against the schema below (part of `lint` when `evals/evals.json` exists).

### Schema (JSON, generated from zod)

```json
{
  "skill_name": "commit-message-writer",
  "settings": {
    "runs_per_prompt": 3,
    "trigger_threshold": 0.8
  },
  "evals": [
    {
      "id": 1,
      "prompt": "Write a commit message for my staged changes",
      "expected_output": "A conventional-commit message generated from the actual staged diff",
      "files": ["evals/files/sample-diff.txt"],
      "expectations": [
        "The output uses a conventional commit type prefix (feat/fix/refactor/docs/test/chore)",
        "The subject line is lowercase and imperative",
        "The message references the renamed function from the diff"
      ]
    }
  ],
  "triggers": [
    { "id": "explicit-1",   "type": "explicit",   "prompt": "/commit-message-writer for the staged diff", "should_trigger": true },
    { "id": "implicit-1",   "type": "implicit",   "prompt": "I staged some changes, write me a good commit message", "should_trigger": true },
    { "id": "contextual-1", "type": "contextual", "prompt": "Just finished the refactor, everything is staged. What should the commit say?", "should_trigger": true },
    { "id": "negative-1",   "type": "negative",   "prompt": "Explain what my last commit changed", "should_trigger": false },
    { "id": "negative-2",   "type": "negative",   "prompt": "Write a PR description for this branch", "should_trigger": false }
  ]
}
```

Validation rules: `skill_name` must equal the skill's resolved name; `evals[].id` unique integers; `expectations` non-empty strings; `files` paths must exist relative to skill root (reuses SC101 resolution); `triggers[].type` in `explicit|implicit|contextual|negative`; `type: negative` requires `should_trigger: false` (and `should_trigger` defaults to `type !== "negative"`); warn when fewer than 2 negative triggers exist (near-miss negatives are the valuable ones); `settings` optional with the defaults shown. Extra unknown keys are allowed and preserved (skill-creator compat).

### Runtime groundwork (interfaces only, no execution in v0.2)

Define a `Driver` interface now: `{ name, run(prompt, skillDir, opts): AsyncIterable<Event>, detectSkillInvocation(events, skillName): boolean }`. Document the three driver realities so v0.3 slots in cleanly:

- **claude-code**: `claude -p --output-format stream-json --verbose --max-turns N`. MUST NOT use `--bare` (bare skips skill discovery); assert skill availability from the `system/init` event. Detect invocation via Skill tool_use events. Since `--bare` will become the `-p` default, plan for `--plugin-dir` injection.
- **opencode**: `opencode run --format json --auto`. Detect via `message.part.updated` tool part with tool name `skill`.
- **codex**: `codex exec --json`. No skill/tool catalog events exist (openai/codex#31088); detection is heuristic -- grep `item.*` command events for the skill's SKILL.md/dir path.

Planned result output (v0.3) mirrors skill-creator `grading.json` (`expectations[{text, passed, evidence}]`, `summary.pass_rate`) and reports trigger results polarity-split (should_trigger and should_not_trigger rates separately -- over-triggering must not hide behind under-triggering).

## 4. SARIF OUTPUT

`--format sarif` (write to stdout; `--output <path>` to write a file). SARIF 2.1.0 only (the only version GitHub accepts). Severity mapping: error -> `"error"`, warning -> `"warning"`, info -> `"note"`. Paths: repo-relative, forward slashes (critical -- path changes for the same result open/close alerts). Omit `partialFingerprints` (upload-sarif computes them). One run per invocation; all 35 rules emitted in `tool.driver.rules` with `defaultConfiguration.level`; every `results[].ruleId` matches a `rules[].id`. `shortDescription.text` <= 255 chars.

```json
{
  "$schema": "https://json.schemastore.org/sarif-2.1.0.json",
  "version": "2.1.0",
  "runs": [
    {
      "tool": {
        "driver": {
          "name": "skillcheck",
          "informationUri": "https://github.com/Sagargupta16/skillcheck",
          "version": "0.2.0",
          "rules": [
            {
              "id": "SC006",
              "name": "name-missing",
              "shortDescription": { "text": "SKILL.md frontmatter must include a name field." },
              "fullDescription": { "text": "The agentskills.io spec requires a name field; skills-ref rejects skills without one." },
              "helpUri": "https://github.com/Sagargupta16/skillcheck/blob/main/docs/rules.md#sc006",
              "defaultConfiguration": { "level": "error" }
            }
          ]
        }
      },
      "results": [
        {
          "ruleId": "SC006",
          "level": "error",
          "message": { "text": "Missing required field in frontmatter: name" },
          "locations": [
            {
              "physicalLocation": {
                "artifactLocation": { "uri": "skills/my-skill/SKILL.md" },
                "region": { "startLine": 1, "startColumn": 1 }
              }
            }
          ]
        }
      ]
    }
  ]
}
```

Limits to respect defensively (never realistically hit): 10 MB gzipped, 25,000 results/run, 20 runs/file. Document the upload snippet in README: `github/codeql-action/upload-sarif@v4` with `category: skillcheck` and `permissions: security-events: write`.

## 5. GITHUB ACTION

**Decision: composite action running `npx`, in a new repo `Sagargupta16/skillcheck-action` (action.yml at repo root -- marketplace requires it).** Not a node24 JS action (would require a bundled `dist/` kept in lockstep with npm -- a second release pipeline a solo builder does not need), not docker. Composite + npx is a proven pattern (reviewdog/action-eslint, withastro/action). If startup latency ever becomes a real complaint, migrate to `using: node24` (already the default JS runtime; node20 dies fall 2026).

**Annotation strategy (three tiers):**
1. Default `--format github` -- the CLI's existing `::error`/`::warning` workflow commands give inline PR annotations for free. Hard caps: 10 errors + 10 warnings + 10 notices per step, 50 per job -- so annotations show the worst N only.
2. Always write the full report as a markdown table to `$GITHUB_STEP_SUMMARY` (CLI does this itself when the env var is present; keeps the composite step dumb).
3. SARIF is opt-in: the action writes the file and exposes its path as an output; the USER wires upload-sarif (uploading inside the action would force `security-events: write` on everyone).

Skip problem matchers entirely (redundant when the CLI emits commands itself).

```yaml
name: "SkillCheck"
author: "Sagar Gupta"
description: "Lint Agent Skills (SKILL.md) for spec conformance with PR annotations and SARIF output"
branding:
  icon: "check-circle"
  color: "purple"
inputs:
  path:
    description: "File or directory to scan recursively for SKILL.md files"
    required: false
    default: "."
  fail-on:
    description: "Minimum severity that fails the check: error | warning | never"
    required: false
    default: "error"
  sarif-file:
    description: "If set, also write a SARIF 2.1.0 report to this path (feed it to github/codeql-action/upload-sarif@v4)"
    required: false
    default: ""
  version:
    description: "@sagargupta16/skillcheck version to run"
    required: false
    default: "0.2.0"   # bump in lockstep with npm publishes
outputs:
  error-count:
    description: "Number of errors found"
    value: ${{ steps.lint.outputs.error-count }}
  warning-count:
    description: "Number of warnings found"
    value: ${{ steps.lint.outputs.warning-count }}
  sarif-file:
    description: "Path to the SARIF report (empty if not requested)"
    value: ${{ steps.lint.outputs.sarif-file }}
runs:
  using: "composite"
  steps:
    - id: lint
      shell: bash
      env:                       # inputs via env, never interpolated into run: (injection safety)
        SC_PATH: ${{ inputs.path }}
        SC_FAIL_ON: ${{ inputs.fail-on }}
        SC_SARIF: ${{ inputs.sarif-file }}
        SC_VERSION: ${{ inputs.version }}
      run: |
        args=(lint "$SC_PATH" --format github --fail-on "$SC_FAIL_ON")
        [ -n "$SC_SARIF" ] && args+=(--sarif "$SC_SARIF")
        npx --yes "@sagargupta16/skillcheck@${SC_VERSION}" "${args[@]}"
```

**CLI prereqs this implies:** `--fail-on error|warning|never`, `--sarif <path>` (sidecar SARIF alongside any format), and self-service `GITHUB_OUTPUT` counts + `GITHUB_STEP_SUMMARY` table when those env vars exist.

**Versioning:** publish full-semver tags (`v0.2.0`) as GitHub Releases via the marketplace flow (immutable releases on). Keep the moving `v1` major tag as a PLAIN git tag never attached to a release (`git tag -fa v1 && git push origin v1 --force`) -- immutable releases lock release tags permanently, so v1 must stay release-free. Marketplace categories: Code quality + Continuous integration.

## 6. CLI DX

Minimal set for a solo maintainer:

- **Rule codes:** `SC###` + kebab alias, interchangeable everywhere (config, suppressions, `--deny/--allow`). ShellCheck uses SC1xxx-SC3xxx 4-digit codes; skillcheck stays 3-digit SC001-SC999, no visual collision. Never recycle retired codes.
- **Exit codes:** the universal contract -- `0` clean (warnings ok under threshold), `1` findings at/above `--fail-on`, `2` config/internal error. Plus `--max-warnings N`. Do NOT conflate crashes with lint failures.
- **Config:** ONE canonical file, `skillcheck.config.json` (jsonc-parsed). Shape: `{ "$schema", "rules": { "SC101" | "broken-relative-reference": "off"|"warn"|"error" }, "ignore": ["globs"], "target": ["spec", "claude-code"] }`. Zero-config must keep working. No rc-file zoo, no `extends`, no nested configs. Generate `configuration-schema.json` from the zod rule registry at build time, ship it inside the npm package (offline `$schema`), file a SchemaStore PR with `fileMatch: ["skillcheck.config.json"]` once stable.
- **No `.skillcheckignore` file** (ESLint v9 killed dedicated ignore files): config `ignore` array + `--ignore-pattern` flag + default `node_modules/.git` excludes + respect `.gitignore` during discovery.
- **--fix scope: none in v0.2.** Tag every rule with `fixable: "safe" | "unsafe" | null` metadata now and print fix-it suggestion text in pretty output (SC021's quote-the-description is the flagship suggestion), but the `--fix` flag itself ships in v0.3. Every competitor's autofix is where their bug reports live; not worth it solo.
- **Suppressions:** file-level only via HTML comment, invisible when rendered: `<!-- skillcheck-disable SC104 broken-relative-reference -->` near the top of SKILL.md. Per-line waits.
- **Output polish:** keep `pretty` default (per-file grouping + code frame + `N fixable` hint footer); add `concise` (`! path:line:col: SC013 (name-invalid-chars): message`); keep `json` and `github`; add `sarif`. `--max-diagnostics 20` cap with "shown X of Y" footer. Honor `NO_COLOR`/`FORCE_COLOR`.
- **`skillcheck init`:** writes `skillcheck.config.json` with `$schema` prefilled. (`eval init` covered in section 3.)
- **npm naming:** check whether unscoped `skillcheck` is free on npm and grab it if so (npx ergonomics: `npx skillcheck` beats `npx @sagargupta16/skillcheck`). Note the ecosystem collision: RiriXt1/skillcheck (17 stars) owns `pip install skillcheck` -- README should self-identify clearly ("skillcheck (TypeScript) -- conformance suite for Agent Skills").
- **Dogfood gate:** CI job lints all 16 skills in the claude-skills marketplace with the new rule set before every release; use `--target claude-code,spec` so extension fields warn instead of error.

## 7. CUT LIST (explicitly NOT in v0.2)

1. **Runtime eval execution** (`skillcheck eval run`, trigger matrix, drivers) -- v0.2 ships scaffold + validate + the Driver interface only. Goal (e) says groundwork; execution is v0.3. This is the differentiating lane (no competitor publishes a real cross-runtime compatibility matrix; agentskills #233 is unclaimed), so do it right, not rushed.
2. **`--fix`** -- metadata + suggestions only (see section 6).
3. **Watch mode** -- cheap but CI never uses it; after fix/SARIF.
4. **0-100 quality scoring / A-F grades** -- competitor gimmick (skill-check, skilldoctor); severity tiers gate CI fine.
5. **Security ruleset beyond SC107** -- prompt-injection/data-exfiltration/destructive-command heuristics (skilldoctor's set) are high-false-positive; revisit with real corpus data.
6. **Description keyword-overlap collision detection** (tripwire/RiriXt1 style) -- needs tuning thresholds; SC105 exact-name duplicates suffice for now.
7. **Remote URL linting, HTML reports, baseline/diff comparison** -- skill-check features with no demand signal.
8. **Per-line suppressions, nested configs, `extends`, `overrides`** -- YAGNI for the target repo shapes.
9. **Node24 JS action / docker action / problem matchers** -- composite npx covers it.
10. **Frontmatter JSON Schema publication** (agentskills #169) -- tempting first-mover win, but it is a spec-org contribution, not a skillcheck feature; also blocked by the OSS contribution pause (no new upstream PRs). Park in TODO for v0.3.
11. **`--since <ref>` changed-only linting, `-D/-A/-W` CLI rule flips, `--staged`** -- nice DX, post-0.2.
12. **skills-ref CLI mirroring** (`read-properties`, `to-prompt`) -- out of scope; skillcheck is a linter, not a reference reimplementation.

Build order recommendation: rules (1) -> SARIF (4) -> CLI DX flags the Action needs (6) -> Action (5) -> eval scaffold (3). Rules and SARIF share the registry work; the Action depends on `--fail-on`/`--sarif`; the eval scaffold is independent and can land last.
